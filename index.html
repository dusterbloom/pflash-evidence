How it Works

The Qwen3-0.6B drafter forward pass at long context dominates PFlash TTFT — 14–16 s on a 32K prompt, 71% of total TTFT in the latency baseline. Empirical ablations showed the drafter’s selection signal was indistinguishable from random or forced-only on niah_multikey_1@16K and CWE@32K (all within 0.2 percentage points).

The DFLASH_SKIP_DRAFTER fast path bypasses this entirely and builds the keep set from three cheap operations:

1. Forced head + tail 8 + 24 chunks always kept. System prompt and recent context are non-negotiable.
2. 4-gram anchor matching Scan body for token n-grams appearing in last 96 query tokens. Force chunk neighborhoods (radius 2) of each match. Selection time: ~0.000 s.
3. Stride-fill Fill remaining keep_ratio budget uniformly across unselected chunks for broad coverage.
Recall floor If anchor_hits == 0 in AUTO mode, fall back to drafter forward. ALWAYS mode uses head/tail+stride without anchor.

The anchor is load-bearing for NIAH single-needle: without it (head/tail + stride only), 16K accuracy dropped to 20% (1/5). With 4-gram anchor matching the question’s repeated keyword, retrieval recovers to 100%.

Why it works: In NIAH single-needle, the question literally repeats the needle’s keyword. The 4-gram anchor matches this verbatim overlap and forces the correct chunk into the keep set. This is a cheap O(body) scan vs a full drafter transformer forward.

How This Is Different

All prior published KV/prefill compression systems share one property: they are unconditional. The selection does not depend on the query text. PFlash skip+anchor breaks this.

SystemMechanismPer-request controlQuery-conditionedAnchor-zero fallback
SnapKV 2404.14469Attention-score gatingNoneNoNone
PyramidKV 2406.02069Layer-wise attn budgetNoneNoNone
H2OHeavy-hitter oracleNoneNoNone
QuestToken quest selectionNoneNoNone
SpecPrefill 2502.02789Drafter-score gatingNoneNoNone (unconditional)
vllm-mlx PR #180Scalar specprefill_keep_pctScalar only, no enumNoNone
PFlash skip+anchor4-gram query-conditioned anchorPFlashMode {OFF, AUTO, ALWAYS} per-request via extra_body.pflash_modeYesanchor_hits==0 → drafter forward (AUTO)

The per-request mode override and the recall floor are not present in any reviewed prior art. The 4-gram scan is O(body length) with no transformer forward.

What Did Not Work

VT (variable tracking) synthetic task

Excluded from all results. The synthetic VT harness in the RULER suite has a filler design that creates spurious n-gram overlap between filler and query, inflating apparent compression quality. The VT cells in the envelope are not reported. This is a harness problem, not a PFlash problem.

pflash_skip_park on RTX 3090 (24 GB)

Works on RTX 3090 (24 GB) WITH ee7 enabled per commit 62bfb11 — server startup is clean, no cuMemSetAccess NOT_READY. NIAH validation pending (script ready at dflash/bench/run_skip_park_32k.py). The historical 24 GB crash was triggered by full-28-layer drafter VRAM footprint, not a hardware limit. Root cause: skip_park left target + drafter simultaneously resident; with ee7’s 7-layer forward, VRAM footprint drops below the VA contiguity fragmentation threshold.

Legacy workaround (full-layer drafter only): GGML_CUDA_NO_VMM=1 or --pflash-skip-park=false. Source: dflash/src/common/model_backend.h:122.

DFlash-chain speculative decode

DFlash-chain (γ=16) produces 11.3% (29/256) acceptance on Claude Code repo-inspection workloads. The Qwen3.6-27B drafter generates tokens that are stylistically mismatched to Claude-formatted completions. MTP γ=2 at 0.79–0.88 is the right pairing with skip+anchor.

Tool-format gap

Qwen3.6-27B-Q4_K_M emits XML-shaped pseudo-tool-calls, not Anthropic tool_use JSON blocks. Claude Code agentic loops end after 1–3 effective turns. This is a chat-template / tool-parser issue, not a PFlash issue. A server-side XML→JSON converter would fix it.

Drafter signal on most workloads

Experiment A v1 (niah_multikey_1@16K, n=50): drafter / random / forced-only all 88.0%, CI [75.7, 95.5]. Experiment A v2 (CWE@32K, n=50, MTP): all within 0.2 pp. The drafter contributes zero measurable value on these tasks. This is why skip+anchor works — but it also means the drafter is worthless for these tasks in any mode.

Q8_0 drafter with ee7 on RTX 3090

Works WITH ee7 — commit f3e3825 shows ~10% speedup vs BF16 at NIAH-equivalent quality. The original “Q8 dead” finding was based on full-28-layer forward; with ee7’s 7-layer forward, dequant cost drops proportionally and BW savings dominate. Q8 reranker drafter is now a documented opt-in. Historical context: full-28-layer Q8_0 on RTX 3090 triggered scalar fallback (0.5× regression vs BF16) — that finding no longer applies under ee7.

Cross-arch drafter: loader ready, BSA kernel not

Speed-ready only for Qwen3-0.6B shape. The loader adapter (commit 7c3df9c, ~30 LOC) accepts qwen3/qwen2/llama-arch GGUFs and SmolLM2-360M loads. But the FlashPrefill BSA kernel rejects non-Qwen3-0.6B head/layer geometry — the fallback path is 4× slower than baseline. Cross-family is loader-ready but not speed-ready until the BSA kernel template is generalized (estimated multi-week CUDA engineering work).

SCORE_LAYERS warm-path regression — FIXED

Fixed in commits f157274 + d3fbad3. Root cause: K_norope_v sized to all 28 layers when ee7 only used 7 → 5.6 GB wasted VRAM → CUDA page migration. Fix: size K_norope_v to n_score_layers (7). Layer-subset is now structurally correct and composes cleanly with ee7. 128K ee7 A_compute = 0.80 s (vs 9.52 s baseline).

Environmental Impact

Faster prefill = fewer GPU watt-hours per request. The mission is explicit: defending and hardening skip+anchor is an environmentally noble goal at inference-server scale.

1.4 Wh saved per 32K request 14 s drafter bypass × 350 W (RTX 3090 peak) = 4.9 kJ = ~1.4 Wh.
~140 Wh/user/day at agentic scale 100 long-context turns/day per user × 1.4 Wh = 140 Wh saved. Roughly equivalent to an LED bulb running 14 hours.
Wall time: 52 s → 4 s at 32K GPU is idle 48 of every 52 seconds in vanilla mode, just waiting on the drafter. skip+anchor reclaims that idle time.
Compounds with MTP γ=2 skip+anchor eliminates the prefill bottleneck; MTP raises decode throughput. Combined: the full request is faster and cheaper.

Architectural Finding — 67% Dead Weight in pflash Mode

In pflash mode the drafter is a positional scorer using only Q&K^T. FFN, LM head, V projection, O projection, and layers 8–28 never execute.

~1.0 GB of the 1.5 GB drafter is dead weight on GPU in pflash mode. A purpose-built scoring drafter would be ~250 MB and would be simultaneously lighter and faster on any hardware.
ComponentSizeUtilization in pflash mode
FFN weights (7 active layers)~130 MB0% (B_compute=0 in profile)
LM output head~310 MB0% (pflash scores positions, does not predict tokens)
V projection~14 MB0% (tail-attention is Q@K^T only)
O projection~7 MB0% (no residual stream output)
Layers 8–28~570 MB0% (ee7 exits after layer 7)
Total dead weight~1.0 GBof 1.5 GB drafter loaded on GPU
