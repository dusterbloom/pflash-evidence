TL;DR

Executive Summary

PR #274 ships two mechanisms stacked on top of ee7: a per-request fa_window_override that widens the target's attention window to cover the full compressed prompt (never caps visibility), and a C2 mechanism gate that picks spec-decode or AR fallback based on effective window size.

At 32K: spec-decode activates, accept rate 14%, +24% tok/J, NIAH recovers 0/3 → 3/3. At 128K: C2 gate correctly selects AR fallback (override exceeds 4096 threshold), composition ties pflash-only on tok/J at verified quality (3/3 NIAH). Drafter wall improves at every context vs ee7 prior d3fbad3.

Known gap: MED-tier (8K–32K) real sessions lose −19% tok/J — drafter VRAM overhead not amortized at that context band. Candidate fix is EMA-driven C2 gate v2.

C2 Mechanism Gate — Design and Honest Gap

per-request flow (PR #274):
  1. pflash drafter scores + compresses prompt
  2. fa_window_override = effective_compressed_tokens + 256   [always widens, never caps]
  3. C2 gate: if fa_window_override <= 2 × cfg_.fa_window (4096):
               → spec-decode (DDTree active)
             else:
               → AR fallback (dFlash token-by-token)
  4. target model runs with fa_window_override set

Why the 2× threshold? Theoretical amortization: at override > 2×cfg.fa_window, the additional attention cost during verification exceeds the draft-skip gain. INFERRED — this threshold comes from analysis, not empirical break-even measurement. The experiment "spec-decode at 128K with C2 gate disabled" has never been run.

What this means in practice

At 32K (override ~1200–1800 tokens, well below 4096×2): gate selects spec-decode. Wins +24% tok/J, accept rate 14%, NIAH 0/3 → 3/3.

At 128K (override ~5621 tokens, above 4096×2): gate forces AR fallback. Composition ties pflash-only on tok/J (−4%, within noise). The drafter VRAM cost is paid but no decode-time gain. This is correct behavior — not a bug — but the threshold is unvalidated empirically.

The gate is a switch, not a knob

No intermediate behavior exists: spec-decode is either fully on or fully off per request. A natural improvement would be variable draft window N — propose fewer tokens per verify at long context, keeping spec-decode active with smaller amortization cost. This is queued as a follow-up (~10 min env-gated experiment, not yet measured).

Open Questions

P2-J: MED tier −18.7% tok/J regression

At 8K–32K real sessions, drafter setup overhead dominates accept-rate gains. Candidate fix: EMA-driven C2 gate v2 that delays spec-decode activation until per-session accept-rate signal justifies the overhead. Not measured; queued as a candidate for v2.

P2-K: C2 gate break-even at 128K unmeasured

The 2× threshold (fa_window_override ≤ 8192 → spec-decode active) is theoretically derived. Running spec-decode at 128K with gate disabled — to empirically measure whether it wins or loses — has never been done. ~10 min experiment.

CLOSED: F1 regression vs 0.628 ceiling

The prior memory entry "F1=0.628 ceiling" was measured on a different stack (FP16 weights, no tq3_0 KV, full 200 cases). Oracle-verified 2026-05-28: expected range for Q4_K_M + tq3_0 on HotpotQA is 55–65%. Our 0.587 (compressed) and 0.547 (uncompressed) both fall inside that range. No regression.

Reproduce

Build

git clone --recurse-submodules https://github.com/Luce-Org/lucebox-hub
cd lucebox-hub
cmake -B server/build -S server -DCMAKE_BUILD_TYPE=Release
cmake --build server/build --target dflash_server -j

Download weights

# Target model (~18 GB)
hf download unsloth/Qwen3.6-27B-GGUF Qwen3.6-27B-Q4_K_M.gguf \
  --local-dir server/models/

# DFlash drafter (decode-phase spec)
hf download Lucebox/Qwen3.6-27B-DFlash-GGUF dflash-draft-3.6-q8_0.gguf \
  --local-dir server/models/draft/

# PFlash drafter (prefill scoring)
hf download Qwen/Qwen3-0.6B-GGUF Qwen3-0.6B-Q8_0.gguf \
  --local-dir server/models/pfill/

Launch (adaptive composition, PR #274)

DFLASH27B_KV_TQ3=1 \
PFLASH_COMPRESS_ANCHOR_TRANSITIVE=1 \
./server/build/dflash_server server/models/Qwen3.6-27B-Q4_K_M.gguf \
  --draft server/models/draft/dflash-draft-3.6-q8_0.gguf \
  --prefill-drafter server/models/pfill/Qwen3-0.6B-Q8_0.gguf \
  --prefill-compression auto \
  --prefill-threshold 32000 \
  --prefill-keep-ratio 0.10 \
  --ddtree --ddtree-budget 22 \
  --fa-window 2048 \
  --port 8000

Run the bench harness

cd bench/2026-05-28_adaptive_stack
python3 run_bench.py --axes A B C

Env Vars Reference

PFlash env vars use PFLASH_* prefix. DFlash env vars use DFLASH_*. Do not mix.

Flag / Env Default Effect
--prefill-compression off When to score+compress: off / auto / always
--prefill-threshold 32000 Token threshold for auto mode
--prefill-keep-ratio 0.05 Fraction of source tokens retained (0.10 for HotpotQA quality)
PFLASH_COMPRESS_ANCHOR_TRANSITIVE 0 Follow transitive anchor edges; required for F1=0.587 quality tier
DFLASH_FP_USE_BSA 0 Sparse FA via BSA kernel (sm_80+); required to enable Block-Sparse Attention dispatch in FlashPrefill
DFLASH_FP_ALPHA 0.85 Block-selection threshold; higher = fewer K-blocks kept
DFLASH27B_KV_TQ3 on TQ3_0 K+V KV cache (3.5 bpv, fits 256K @ 24 GB)
DFLASH_FP_PROFILE 0 Per-stage timing log to stderr
--prefill-skip-park off Keep drafter resident across requests. Requires ≥32 GB VRAM
